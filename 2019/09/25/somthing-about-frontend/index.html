干啥呢

总结一下前端开发中常用的知识。

计算机网络相关

输入 url 后发生了什么

  1. 输入地址
  2. DNS解析
  3. TCP连接
  4. 发送http请求
  5. 返回http响应
  6. 浏览器解析渲染页面 https://www.cnblogs.com/slly/p/6640761.html
    • DOM Tree -> CSS Rule Tree -> rendering tree -> layout -> painting
  7. 断开连接

POST vs. GET

  1. GET 主动 cache
  2. GET 有长度限制
  3. GET 只能 url 编码
  4. GET只接受 ASCII 字符，POST没有限制
  5. GET把数据包含在URL中，POST通过request body传递数据

  6. GET在浏览器回退时是无害的，POST会再次提交请求

  7. GET请求在URL中传送有长度限制，POST没有

http vs. https

  • http：被监听、被篡改、被伪装
  • https：解决上述问题

RESTful API

  • 无状态
  • 对于资源的操作类型由HTTP动词来表示

TCP vs. UDP

  • 可靠 不可靠
  • 基于连接 无连接
  • 流模式 数据报模式

http1.1 vs. http2 vs. http3(quic)

  • http1.1
    • Connection:keep-alive
  • http2
    • 二进制传输（之前是纯文本）
    • HEADER 压缩
    • 多路复用
      • 同域名下所有通信都在单个连接上完
      • 双向数据流
      • 数据流以消息的形式发送，而消息又由一个或多个帧组成，多个帧之间可以乱序发送，因为根据帧首部的流标识可以重新组装
  • http3(quic)
    • UDP，解决 http2 中的队头阻塞

长连接 vs 短连接

没有数据传输时，长连接不关闭连接

xss 攻击

Cross Site Scripting

  • 反射型
  • 存储型

防范：

  • 入参字符过滤：把输入的一些不合法的东西都过滤掉，从而保证安全性。如移除用户提交的的DOM属性如onerror，移除用户上传的Style节点
  • 出参进行编码：&gt; 代替 >
  • cookie 设置为 httponly：通过程序(JS脚本、Applet等)将无法读取到Cookie信息，这样能有效的防止XSS攻击。

CSRF 攻击

Cross Site Request Forgery

  1. 验证HTTP Referer字段
  2. 添加token验证（正常操作时随机生成，服务端验证。被伪装的时候没有/错误）
  3. 验证码

WebSocket

一种协议，没有同源限制，双向通信

浏览器相关

cookie vs. session vs. localStorage vs. sessionStorage

  • cookie: 浏览器端
  • session: server side

  • localStorage: 永久，同一浏览器的相同域名和端口的不同页面间可以共享

  • sessionStorage: 关闭浏览器

浏览器的同源策略

同源：协议+网址+端口

repaint vs. reflow (重排/回流)

重绘不一定重排

重排一定重绘

load 与 DOMContentLoaded

  • load：的html、css、js、图片等资源都已经加载完之后才会触发 load 事件。
  • DOMContentLoaded（先）：意思是HTML下载、解析完毕之后就触发

JavaScript

闭包

  • usage
    • 匿名自执行函数
    • 结果缓存
  • 优点
    • 可以读取函数内部的变量
    • 可以让这些局部变量保存在内存中，实现变量数据共享。
  • 缺点
    • 滥用：内存消耗很大

原型

  • 实例的__proto__属性（原型）等于其构造函数的 prototype属性

  • 构造函数不需要显示的返回值。使用new来创建对象，调用构造函数时，如果return的是非对象(数字、字符串、布尔类型等)会忽而略返回值;如果return的是对象，则返回该对象(注：若return null也会忽略返回值）

  • 通过原型链继承

事件冒泡 vs. 事件捕获 vs. 事件委托

  • 事件冒泡：子 -> 父
  • 事件捕获：父 -> 子， `addEventListener(xxx, xxx, true)
  • 事件委托 ：委托它们父级代为执行事件

React

组件生命周期

  • componentWillMount 在渲染前调用,在客户端也在服务端。
  • componentDidMount : 在第一次渲染后调用，只在客户端。之后组件已经生成了对应的DOM结构，可以通过this.getDOMNode()来进行访问。 如果你想和其他JavaScript框架一起使用，可以在这个方法中调用setTimeout, setInterval或者发送AJAX请求等操作(防止异步操作阻塞UI)。
  • componentWillReceiveProps 在组件接收到一个新的 prop (更新后)时被调用。这个方法在初始化render时不会被调用。
  • shouldComponentUpdate 返回一个布尔值。在组件接收到新的props或者state时被调用。在初始化时或者使用forceUpdate时不被调用。
    可以在你确认不需要更新组件时使用。
  • componentWillUpdate在组件接收到新的props或者state但还没有render时被调用。在初始化时不会被调用。
  • componentDidUpdate 在组件完成更新后立即调用。在初始化时不会被调用。
  • componentWillUnmount在组件从 DOM 中移除之前立刻被调用。

why 虚拟 dom

虚拟 dom 相当于在 js 和真实 dom 中间加了一个缓存，利用 dom diff 算法避免了没有必要的 dom 操作，从而提高性能。

用 JavaScript 对象结构表示 DOM 树的结构；然后用这个树构建一个真正的 DOM 树，插到文档当中当状态变更的时候，重新构造一棵新的对象树。然后用新的树和旧的树进行比较，记录两棵树差异把 2 所记录的差异应用到步骤 1 所构建的真正的 DOM 树上，视图就更新了。

React diff

tree diff

  • React 通过 updateDepth 对 Virtual DOM 树进行层级控制，每次 diff 进行同层比较

  • 当出现节点跨层级移动时，会以 A 为根节点的树被整个重新创建

component diff

  • 如果是同一类型的组件：按照原策略继续比较 virtual DOM tree
  • 如果不是：不会比较二者的结构，直接删除旧组件，重新创建新组件以及其子节点

element diff

  • 当节点处于同一层级时，React diff 提供了三种节点操作，分别为：INSERT_MARKUP（插入）、MOVE_EXISTING（移动）和 REMOVE_NODE（删除）
